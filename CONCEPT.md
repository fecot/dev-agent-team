# CONCEPT — 開発の型を作るとはどういうことか

## 問題意識

SaaS開発の現場で感じることがある。

優秀なエンジニアでも、タスクが積まれ余裕がなくなると「とりあえず動くものを書く」モードに入る。要件を口頭で確認した気になり、影響範囲を確認せずに実装し、テストを省き、PRの説明を「Fix #123」で済ませる。個々の選択は理解できるが、積み重なると負債になる。

問題は「個人の能力」ではなく「プロセスの欠如」だ。型がなければ、忙しい状況では人は省略する。

## 解決の方向

Claude Codeを「コードを書く道具」として使うのではなく、**開発プロセスを伴走する存在として設計する**。

具体的には:

1. **イシューを受け取ったら、まず要件を整理する**（ProductInterpreter）
2. **既存コードを読んで、文脈を把握する**（CodebaseExplorer）
3. **変更の影響範囲を分析する**（ArchitectureReviewer）
4. **実装案を複数出し、最小案も含めて選ぶ**（ImplementationDriver）
5. **テスト観点を明示する**（TestStrategist）
6. **PRを説明可能な状態にする**（ReviewGatekeeper）
7. **リリースリスクを確認する**（ReleaseCaptain）

各フェーズをエージェントとして定義することで、Claude Codeのサブエージェント呼び出しや、スラッシュコマンドとして再現性のある手順を作れる。

## 型とは何か

「型」とは制約ではなく、品質の下限を保証するためのチェックリストと問いの集合体だ。

熟練エンジニアは型を意識せず踏んでいる。そのプロセスを明文化し、Claude Codeに実行させることで、経験に関わらず一定の水準を担保できる。

## AIにやらせること、人間がやること

AIにやらせること:
- 既存コードの調査と要約
- 影響範囲の洗い出し
- 実装案の列挙
- テストケースの網羅
- PR説明文の生成

人間がやること:
- 要件の最終確認と判断
- 設計の意思決定
- リリースの承認

AIは「抜け漏れを防ぐ副操縦士」として機能させる。主体は常に人間だ。

## 型とエンジンは別レイヤー

Claude Code 本体は `/goal`（自律ループ）や Dynamic Workflows（並列オーケストレーション）といった **実行エンジン** を提供する。これらが出てきたことで「dev-agent-team はもう不要では?」と思うかもしれないが、レイヤーが違う。

- **dev-agent-team が出すのは「何を・なぜ」**——プロセス規律、Stop Condition、Human Decision Point、移行/UI/レガシーのチェックリスト。つまり **意思決定の型** だ。
- **エンジンが出すのは「どう動かすか」**——速く・広く・収束するまで回す実行機構だ。エンジンはあなたのプロセスを知らない。

両者は競合せず積み重なる。dev-agent-team は **自分の判断の型を、より強力なエンジンで実行する** ために `/goal` や Dynamic Workflows を取り込む。ただし「判断は人間」という原則は変えない——エンジンには **人間ゲートを越えさせない**。具体的な併用ルールは [`docs/native-tooling-integration.md`](docs/native-tooling-integration.md) に定義する。
