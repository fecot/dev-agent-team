# サンプルイシュー: ユーザー一覧にメールアドレス検索機能を追加する

このファイルは、`/issue-to-plan` コマンドの入力となるイシュー例と、各エージェントがどう処理するかのウォークスルーです。

---

## 入力: イシュー

```
タイトル: ユーザー一覧にメールアドレスで絞り込む検索機能を追加してほしい

担当者からの依頼:
CSチームから「特定ユーザーを探すのに時間がかかる」という声があった。
ユーザー一覧画面でメールアドレスを入力して絞り込めると助かるとのこと。
```

---

## Step 1: ProductInterpreter の処理

### 要件整理

**背景・目的**
CSチームが特定ユーザーを探す作業に時間がかかっている。メールアドレスで絞り込むことで検索時間を短縮する。

**やること**
- ユーザー一覧画面にメールアドレス検索フォームを追加
- 入力値でユーザーリストをフィルタリング
- 検索結果が0件の場合の表示

**やらないこと**
- 名前・電話番号など他フィールドでの検索（今回はメールのみ）
- 検索条件の保存・履歴機能
- ページネーションの変更

**完了条件**
- [ ] メールアドレスを部分一致で検索できる
- [ ] 大文字小文字を区別しない
- [ ] 0件のとき「該当するユーザーが見つかりません」と表示される

**確認事項（要確認）**
- フロントエンドのみでフィルタリングする？それともAPIにクエリパラメータを追加する？
- ユーザー数が多い場合のパフォーマンス要件はあるか（現在何件？）

## Step 2: CodebaseExplorer の処理

```
# コードベース調査

## 関連ファイル

## 類似実装

フロントエンドのみで filter() を使う方式。
参考にできる。

## 命名規則
- 検索: `search` ではなく `filter` を使っている
- 状態: useState で `filterQuery` という命名
```

## Step 3: ArchitectureReviewer の処理

```
# 影響範囲

## 変更対象
- pages/users/index.tsx
- components/UserTable.tsx（フィルタリングロジック追加）

## 影響なし
- api/users.ts（フロントエンドのみの場合）
- types/user.ts

## リスク
| リスク | 深刻度 | 対応方針 |
|-------|-------|---------|
| ユーザー数が多い場合の描画パフォーマンス | 中 | useMemo でメモ化 |

## 確認事項
ユーザー数が1000件超える場合はAPIでフィルタリングを検討すること。
今回はフロントのみで実装し、必要なら後でAPIに移行する方針を提案する。
```

---

## Step 4: ImplementationDriver の出力（概要）

**案A（最小変更）**: UserTable にフィルタ用 input を追加し、`filter()` で絞り込む。API変更なし。

**案B（標準）**: フック（useUserFilter）を切り出して再利用可能にする。

今回は案Aを採用。ユーザー数が少ない現状ではシンプルな実装を優先する。

---

## Step 5: TestStrategist の出力（概要）

**必須テスト**
- 部分一致で絞り込まれること
- 大文字小文字を区別しないこと
- 0件のとき空状態メッセージが表示されること
- 入力クリアで全件表示に戻ること
