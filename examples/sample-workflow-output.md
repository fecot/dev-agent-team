# Sample Workflow Output: メールアドレス検索の追加

このファイルは、[`sample-issue.md`](sample-issue.md) のIssueに対して `/run-feature-workflow` を実行したと **仮定** した場合の出力サンプルです。

> ⚠️ **これは実装ログではなく、開発プロセスのデモです。**
>
> - 実在するアプリケーションコードはありません
> - 各 Phase に登場するファイル名は **「想定される調査対象」** として書いてあるだけで、実在を保証するものではありません
> - 実装ステップは「こう進めるだろう」という想定ログであり、実コードは存在しません
> - 各 Phase の Output / Stop Condition Check / Human Decision Required の **書き方の型** を示すための教材です

- Issue ID（仮）: `email-search`

### Phase 1: Intake

- **Input**:
  - `examples/sample-issue.md`（Title / Background / Request / Expected Behavior / Constraints / Acceptance Criteria / Notes）
- **Actions**:
  - ProductInterpreter として要件を整理
  - 「やること」「やらないこと」「完了条件」を Issue 本文から抽出
  - 不明点を確認事項としてリスト化
- **Findings**:
  - 検索対象は **メールアドレスのみ**（Notes に明記）
  - 検索条件の保存・履歴機能は **やらない**（Notes に明記）
  - パフォーマンス要件・実装方式（フロント完結 or API）は未確定
- **Output**:
  - 要件整理ドキュメント `.dev-agent-team/requirements/email-search.md`
  - 内容:
    - **やること**: 一覧画面にメールアドレス検索フォームを追加 / 部分一致 / 大小区別なし / 0件時の空状態 / 入力クリアで全件復帰
    - **やらないこと**: 名前・電話番号など他フィールド検索 / 検索条件保存 / ページネーション挙動の変更
    - **完了条件**: Acceptance Criteria 5項目をそのまま転記
    - **確認事項**:
      1. フロントエンドのみで `filter()` するか、API にクエリパラメータを追加するか
      2. 現在のユーザー件数・想定上限・パフォーマンス要件
      3. 既存に類似の絞り込み実装はあるか（命名規則・パターンを踏襲したい）
      4. 検索結果0件時のメッセージ文言は既存の空状態 UI と揃えるべきか
- **Risks**:
  - 確認事項1〜2 の回答次第で実装方式が大きく変わる
- **Stop Condition Check**:
  - 要件が曖昧 → **Fail**（実装方式とパフォーマンス要件が未確定）
  - やらないこと未定義 → Pass
  - 確認事項への回答 → 未取得
- **Human Decision Required**:
  - 確認事項4項目への回答
- **Next**: 確認事項への人間からの回答待ち。回答取得後 Phase 2 へ。

> このサンプルでは「フロントのみで実装、ユーザー数は当面1万件以下、類似実装ありそう、空状態文言は既存 UI に合わせる」という回答を得たと仮定して以降を続ける。

### Phase 2: Discovery

- **Input**:
  - Phase 1 の要件整理ドキュメント
  - 人間からの確認事項回答（フロント完結 / 1万件以下 / 類似実装ありそう / 空状態文言は既存に合わせる）
- **Actions**:
  - CodebaseExplorer として、ユーザー一覧画面・類似する絞り込み実装・空状態 UI の所在を `grep` / `Glob` で調査
  - データモデル（User 型・API レスポンス）と、検索フォーム共通コンポーネントの有無を確認
- **Findings**（**すべて想定される調査対象。実在は未確認**）:
  - ユーザー一覧画面: `pages/users/index.*` 付近にあると想定
  - ユーザー一覧テーブル: `components/UserTable.*` のような命名で存在しうる
  - 取得フック: `hooks/useUsers.*` のようなカスタムフックを想定
  - 型定義: `types/user.*` 付近に `User` 型を想定
  - 類似実装候補: 商品一覧・組織一覧など、別リソースの一覧画面で既に絞り込みフォームが存在する可能性が高い
  - 空状態 UI: 共通コンポーネント（`components/EmptyState.*` のようなもの）が存在する可能性
- **Output**:
  - 調査レポート `.dev-agent-team/reports/investigation-email-search.md`
  - 「想定される調査対象」のリストと、確認事項として残った疑問
- **Risks**:
  - 類似実装が存在しない場合は、命名規則・パターンを Phase 4 で新規に決める必要がある
  - 一覧データのフェッチ方式（クライアントサイドで全件持つ vs ページネーションごとに API）が不明な場合、フロントのみのフィルタが成立しない可能性
- **Stop Condition Check**:
  - 関連ファイル特定 → **暫定 Pass**（候補列挙済み、実在確認は実コードにアクセスする実環境で行う前提）
  - 類似実装の有無 → **Conditional Pass**（候補は挙げたが、実在は実環境で要確認）
  - データモデル把握 → 暫定 Pass
- **Human Decision Required**:
  - なし（このサンプルでは想定値で進める）。実環境では「候補ファイルが実在するか・データ取得方式」の確認が必要
- **Next**: Phase 3 へ。

### Phase 3: Impact Analysis

- **Input**:
  - Phase 1 の要件整理 / Phase 2 の調査レポート
- **Actions**:
  - ArchitectureReviewer として、変更が各レイヤーに与える影響を観点別に整理
- **Findings**:

  | 観点 | 影響 | 深刻度 | 備考 |
  |---|---|---|---|
  | UI | あり | 低 | 検索フォーム追加・テーブルへのフィルタ反映 |
  | API | なし（暫定） | — | フロント完結方針なら API 変更なし。既存取得 API の挙動は変えない |
  | DB | なし | — | スキーマ変更・マイグレーション不要 |
  | バッチ | なし | — | 一覧画面のみの変更 |
  | 外部連携 | なし | — | |
  | 権限 | なし | — | 既存ロールのまま、追加の認可定義なし |
  | テスト | あり | 中 | 一覧画面のテストにフィルタ系ケース追加。既存ページネーションとの併用ケース必須 |

- **Output**:
  - 影響範囲レポート `.dev-agent-team/reports/impact-email-search.md`
- **Risks**:
  - ユーザー件数が想定（1万件以下）を超えた場合の描画パフォーマンス（深刻度: 中）
  - 既存ページネーションとフィルタの併用ロジックの設計ミスによる挙動破綻（深刻度: 中）
- **Stop Condition Check**:
  - DB変更ありなのにマイグレーション戦略未検討 → N/A（DB変更なし）
  - 権限変更ありなのに既存ロール影響未確認 → N/A（権限変更なし）
  - 外部連携への影響未検証 → N/A
  - 「影響なし」と断言できる根拠がない → **Conditional Pass**（フロント完結方針が前提。方針変更時は再評価）
- **Human Decision Required**:
  - なし。ただし「ユーザー件数が想定を超えたら API 方式に切り替える」を運用判断として明記しておく
- **Next**: Phase 4 へ。

### Phase 4: Implementation Planning

- **Input**:
  - Phase 1〜3 の成果物
- **Actions**:
  - ImplementationDriver として、実装案を **3案** 提示（最小変更案を必ず含める）
  - `templates/implementation-plan-template.md` の形式で計画を起こす
- **Findings**:

  #### 案A（最小変更）: フロントエンドのみで `filter()` 絞り込み
  - **概要**: 一覧画面のステートに `filterQuery` を追加し、取得済みユーザー配列を `filter()` で絞り込む
  - **変更想定ファイル**: ユーザー一覧画面 + 一覧テーブルコンポーネント（合計2ファイル想定）
  - **メリット**: API 変更なし / 差分最小 / ロールバックは PR Revert のみで完結
  - **デメリット**: 件数が増えると描画コスト増。ページネーションとの併用ロジックを丁寧に設計する必要あり
  - **適用条件**: ユーザー件数が当面1万件以下である前提
  - **ロールバック容易性**: 高（フロント差分のみ）

  #### 案B（標準）: フィルタフックを切り出して再利用可能にする
  - **概要**: `useFilteredList` のような汎用フックを切り出し、メールアドレス以外でも再利用可能にする
  - **変更想定ファイル**: 一覧画面 + テーブル + 新規フック1つ（合計3〜4ファイル想定）
  - **メリット**: 将来の絞り込み追加（名前・電話番号など）が容易
  - **デメリット**: 今回スコープ外の汎用化が混入する。再利用ニーズが本当にあるか未確定
  - **適用条件**: 他の一覧でも同じフィルタ機構を使う計画が既にある場合
  - **ロールバック容易性**: 中（共通モジュール経由のため影響線を追跡）

  #### 案C（API拡張）: クエリパラメータでサーバ側絞り込み
  - **概要**: ユーザー取得 API に `email` クエリパラメータを追加し、サーバー側で絞り込んで返す
  - **変更想定ファイル**: フロント + API ハンドラ + テスト（合計5ファイル以上想定）
  - **メリット**: ユーザー件数が大きく増えても性能劣化しにくい
  - **デメリット**: 影響範囲が API・テストにまで広がり、今回のスコープを超える
  - **適用条件**: 件数が10万件オーダー以上、もしくはサーバー側での監査ログ要件がある場合
  - **ロールバック容易性**: 低（API スキーマ変更を伴う）

- **Output**:
  - 実装計画ドキュメント `.dev-agent-team/plans/implementation-plan-email-search.md`
  - 採用判断結果（人間決定後に記録）
- **Risks**:
  - 案Aを採用しても、想定件数を超えた場合は将来 API 方式へ移行する必要が生じる
- **Stop Condition Check**:
  - 実装案が1案しかない → Pass（3案提示済み）
  - 最小変更案がない → Pass（案Aが最小変更）
  - 採用案を人間が決定したか → **未決定（次に人間判断）**
  - ロールバック手順が定義されているか → Pass
- **Human Decision Required**:
  - 案A / 案B / 案C のうちどれを採用するか
  - 「件数が想定を超えた場合は API 方式に切り替える」運用ルールの合意
- **Next**: 人間が採用案を決定したら Phase 5 へ。

> このサンプルでは「**案A 採用**。理由: ユーザー件数の現状規模ではシンプルさを優先。汎用化は実需が出てから検討」と決定したと仮定する。

### Phase 5: Safe Implementation

- **Input**:
  - Phase 4 で採用された案A
- **Actions**:
  - ImplementationDriver として、計画書のステップ順に実装する想定ログを生成
  - 1ステップごとにテストを通す前提
- **Findings**（**実コードは生成していない、想定ログ**）:

  | コミット | 内容 | 確認 |
  |---|---|---|
  | 1 | `filterQuery` ステート追加（初期値: 空文字） | ステート初期値の単体テスト |
  | 2 | 検索フォーム UI を追加（ラベル・アクセシビリティ対応含む） | レンダリングテスト |
  | 3 | テーブル描画時に `filter()` で絞り込みロジックを適用 | 部分一致・大小区別なしのテスト |
  | 4 | 0件時に既存空状態コンポーネントを呼び出す | 空状態のレンダリングテスト |
  | 5 | 入力クリアで全件復帰 | クリア操作のテスト |
  | 6 | ページネーションとの併用挙動を調整 | ページネーション併用テスト |

- **Output**:
  - 実装ログ `.dev-agent-team/logs/implementation-email-search.md`
  - 計画外発見事項（このサンプルでは「なし」と仮定）
- **Risks**:
  - 既存の検索フォーム共通コンポーネントの仕様確認が必要（名前付きスロット・バリデーション仕様など）
- **Stop Condition Check**:
  - 計画にない大きな変更 → なし（Pass）
  - テスト未通過のまま進行 → 想定では各コミットで通過（Pass）
  - 機能追加とリファクタリングの混在 → なし（Pass）
  - 差分が大きすぎる → 6コミット規模、レビュー可能（Pass）
- **Human Decision Required**:
  - 計画外の事項が出た場合のみ。本サンプルではなし。
- **Next**: Phase 6 へ。

### Phase 6: Test Design

- **Input**:
  - Phase 1 の完了条件
  - Phase 5 の実装ログ
  - Phase 3 の影響範囲
- **Actions**:
  - TestStrategist として、完了条件をテストケースに分解しレイヤを割り当てる
- **Findings** — 完了条件 → テストケースのトレーサビリティ:

  | 完了条件 | 観点 | テストケース | レイヤ |
  |---|---|---|---|
  | 部分一致で絞り込まれる | 正常系 | `john@` 入力で `john@example.com` がヒット | 単体（コンポーネント） |
  | 大小区別なし | 正常系 | `JOHN@` 入力でも `john@example.com` がヒット | 単体 |
  | 0件時に空状態 | 境界値 | 該当しない文字列入力で空状態メッセージ表示 | 単体 |
  | 入力クリアで全件復帰 | 回帰 | 入力後にクリアすると全件再表示 | 単体 |
  | ページネーション併用で破綻しない | 回帰 | フィルタ中にページ送り操作しても挙動が壊れない | 結合 |
  | 空入力時は全件表示 | 境界値 | 初期状態で全件表示される | 単体 |
  | 不正な入力（空白のみ・記号のみ） | 異常系 | 空白のみ・記号のみの入力で例外を投げない | 単体 |

- **Output**:
  - テスト計画ドキュメント `.dev-agent-team/plans/test-plan-email-search.md`
- **Risks**:
  - E2E レイヤでの検証要否は既存のテスト運用次第（未確認なら確認事項として残す）
- **Stop Condition Check**:
  - 完了条件と紐づかないテストしかない → Pass（全完了条件にテストが対応）
  - 異常系・境界値の観点欠落 → Pass（含まれている）
  - 影響範囲のうち未テスト領域 → Pass（ページネーション併用の回帰を含む）
- **Human Decision Required**:
  - E2E テストの追加可否（プロジェクト方針による）
- **Next**: Phase 7 へ。

### Phase 7: Review Gate

- **Input**:
  - Phase 1〜6 の成果物すべて
- **Actions**:
  - ReviewGatekeeper として、PR 説明文を `templates/pr-description-template.md` 形式で生成
  - DB変更・権限変更・破壊的変更の有無をチェック
  - セキュリティ観点（OWASP Top 10）でセルフレビュー
- **Findings**:
  - DB変更・権限変更・API契約変更: いずれもなし
  - セキュリティ: 入力値はクライアントの絞り込みのみで使用、SQLi・XSS・SSRF いずれにも該当しない
  - パフォーマンス: 1万件以下の前提では `useMemo` でメモ化すれば許容範囲
- **Output**:
  - レビューノート `.dev-agent-team/reviews/pr-review-email-search.md`
  - PR 説明文 `.dev-agent-team/reviews/pr-description-email-search.md`（後段の「PR説明文サンプル」参照）
- **Risks**:
  - 既存の検索フォーム共通コンポーネントを使えるか / 別途追加するか の差分量がレビューで論点になる可能性
- **Stop Condition Check**:
  - DB / 権限変更ありなのに観点未明記 → N/A
  - テスト観点が PR 説明から欠落 → Pass（テスト計画を要約して記載）
  - 計画と実装のズレが説明されていない → Pass（ズレなし）
  - 「Fix #123」しか書かれていない PR → Pass（テンプレート使用）
- **Human Decision Required**:
  - PR レビュアーの指名・マージ可否判断
- **Next**: Phase 8 へ。

### Phase 8: Release Check

- **Input**:
  - Phase 4 の実装計画（リスク・ロールバック手順）
  - Phase 7 で承認された PR
- **Actions**:
  - ReleaseCaptain として、リリース手順 / 監視観点 / ロールバック判断基準を整理
- **Findings**:
  - リリース方式: 通常リリースで可（フィーチャーフラグ不要）。DB変更がないため段階リリースの必要性は低い
  - 監視観点:
    - ユーザー一覧画面のレンダリング時間メトリクス
    - エラーレート（フロントエンド例外）
    - CSチームからのフィードバック（実利用での体感）
  - ロールバック手順: PR Revert で即時復旧可能。データ移行を伴わないため副作用なし
  - ロールバック判断基準:
    - レンダリング時間が現状比 +50% を超えた状態が10分以上継続したら revert
    - 一覧画面でフロントエンド例外率が現状比 +1ポイント超過で10分以上継続したら revert
- **Output**:
  - リリースチェックリスト `.dev-agent-team/releases/release-checklist-email-search.md`
- **Risks**:
  - リリース後にユーザー数が想定を超えた場合のパフォーマンス劣化（事前に閾値で監視）
- **Stop Condition Check**:
  - DB変更でマイグレーション順序未確認 → N/A
  - ロールバック手順が実行不可能 → Pass（PR Revert で完結）
  - 監視観点未定義 → Pass
  - リリース可否を人間が判断していない → **未決定**
- **Human Decision Required**:
  - リリース可否の最終判断
  - リリース日程（次のリリースサイクル）
- **Next**: 人間がリリース可否を判断したらデプロイ。

## PR説明文サンプル

このサンプルでは Phase 7 の Output として、以下の PR 説明文が生成される想定。

```markdown
## 変更概要
ユーザー一覧画面にメールアドレスでの絞り込み検索を追加する。

## 変更理由
CSチームから「特定ユーザーを探すのに時間がかかる」という声が複数寄せられていた。
問い合わせ対応中にアカウント特定が高速化されることで、対応時間の短縮を見込む。

## 変更内容
- ユーザー一覧画面の上部にメールアドレス検索フォームを追加
- 入力値で取得済みユーザーリストをフロント側で部分一致フィルタ（大小区別なし）
- 0件時は既存の空状態コンポーネントを表示
- 入力クリアで全件復帰
- ページネーションとの併用挙動を調整

## テスト
- 部分一致 / 大小区別なし / 0件 / クリア復帰 / 空入力 / 不正入力（空白・記号）の単体テスト
- ページネーション併用の結合テスト

## 影響範囲
- UI: ユーザー一覧画面、テーブルコンポーネント
- API: 変更なし
- DB: 変更なし
- 権限: 変更なし
- 外部連携: 変更なし

## ロールバック手順
本 PR を revert する。データ移行を伴わないため即時復旧可能。

## 確認してほしい点
- 既存の検索フォーム共通コンポーネントを再利用すべきか、画面固有に置くべきか
- ユーザー件数が想定（1万件以下）を超えるユースケースがある場合、API 方式（案C）への移行プランを別 Issue で起こすべきか
```
