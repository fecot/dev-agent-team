# Issue Intake Template

## 目的

依頼者しか知らない情報を確実に渡すための型です。

- **書かなくて OK な情報**: コードを読めば分かること（既存実装の touch-point / 命名規則 / 影響範囲 / 既存パターン）。これらは Phase 2 Discovery で AI が掘ります
- **書いてほしい情報**: 依頼者の頭の中にしかないこと（受け入れ基準 / 形容詞の数値 / 既知の罠 / 共通部品の期待挙動 / データの実態）

「全部書く」ではなく「**前者を省いて後者に集中する**」のがポイントです。Phase 1 Intake で AI が後追い質問する往復を減らし、Phase 2 以降の Discovery の質を上げることが目的。

## 使い方

1. `/run-feature-workflow` を起動する前に、**必須セクション**（背景・目的 / スコープ / 受け入れ基準）を埋める
2. **該当時のみ必須セクション** は、当てはまるブロックだけ埋める（残りは削除 OK）
3. **任意セクション** は書ければ書く
4. 埋め切れない項目は `未確定（要確認）` と明示する（空欄禁止）
5. 起動時にこのテンプレを丸ごと `/run-feature-workflow` に投入する、または GitHub Issue / Jira チケットの本文として貼り付ける

> 未記入で起動した場合は Phase 1 でテンプレを提示して埋めるよう促されます。Stop Condition に該当するため、空のまま Phase 2 に進むことはありません。
